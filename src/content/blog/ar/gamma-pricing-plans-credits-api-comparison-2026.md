التسعير في Gamma لا يُفهم من سعر الاشتراك فقط، بل من **الاعتمادات الشهرية + معدل الخصم الفعلي + مسار API**.

> تاريخ البيانات: 2026-04-02  
> ملاحظة: القيم والأسعار تتغير حسب المنطقة والتحديثات. عند أي تعارض، اعتمد [Gamma Pricing](https://gamma.app/pricing).

## 1) الخطط: الاعتمادات والفروق العملية

فروقات Free/Plus/Pro/Ultra/Teams/Business تظهر في:

- الاعتماد الشهري
- سقف الشرائح في كل توليد
- توفر API
- ميزات الإدارة والتعاون في الفرق

وللتقييم السريع، استخدم هذا الملخص:

| الخطة | الاعتماد الشهري (حسب الوثائق) | أهم فرق عملي |
| --- | --- | --- |
| Free | 400 لمرة واحدة للمستخدم الجديد | للتجربة المحدودة |
| Plus | 1,000 | بداية الاستخدام المستمر |
| Pro | 4,000 | وصول API وسقوف أعلى |
| Ultra | 20,000 | حمل مرتفع وتوليد مكثف |
| Teams | 6,000 لكل مقعد | إدارة فريق وفوترة مركزية |
| Business | 10,000 لكل مقعد | خصائص مؤسسية أوسع |

### 1.1 الترحيل والشراء الإضافي

- يمكن ترحيل الرصيد غير المستخدم حتى حد يقارب 2× المنحة الشهرية
- تتوفر حزم شراء إضافية للمشتركين المدفوعين
- الخطة المجانية لا تمنح تحديثًا شهريًا للاعتمادات

## 2) لا تخلط بين منطق UI ومنطق API

### 2.1 الاستهلاك داخل التطبيق

المعدلات المذكورة في مركز المساعدة (مثل Create/Add card/Chat/Image) مفيدة للتقدير السريع، لكنها ليست بديلًا عن قياسك الفعلي.

| العملية | الاستهلاك المرجعي |
| --- | --- |
| Create with AI | 40 لكل Gamma |
| Add card with AI | 5 لكل بطاقة |
| Chat with AI | 10 لكل اقتراح مفيد |
| AI image prompt | 10 لكل Prompt |
| Continue with AI | 2 لكل استخدام |

### 2.2 الاستهلاك عبر API

في API، مرجع الحقيقة هو `credits.deducted` في استجابة الجيل.  
الخصم لكل بطاقة أو صورة يأتي كنطاق وليس رقمًا ثابتًا، ويتغير حسب النموذج والإعدادات.

المنهج الصحيح: ابدأ بنطاق تقديري من الوثائق، ثم عاير أرقامك على بيانات التشغيل الحقيقية.

## 3) كيف تحسب تكلفة "عرض واحد"

- استخدام الويب فقط: ابدأ من تكلفة Create ثم أضف بطاقات AI والصور والدردشة
- دفعات API: استخدم نطاقات الوثائق ثم صحّحها ببيانات `credits.deducted`
- Remix/Template: احتسب هامشًا أعلى لأن الاستهلاك قد يزيد

مثال سريع: إذا أنشأت 25 عرضًا شهريًا عبر Create فقط، فأنت تقترب من 1,000 اعتماد (حد Plus تقريبًا) قبل أي بطاقات إضافية أو صور.

## 4) الحسابات المشتركة مقابل المقاعد الرسمية

في العادة، الاعتمادات تُدار على مستوى المستخدم، لا كحوض موحد للفريق بالكامل.  
مشاركة تسجيل دخول واحد قد تناسب اختبارًا قصيرًا، لكنها أضعف في التتبع والمساءلة وحوكمة الوصول.

المخاطر الشائعة للحساب المشترك:

- صعوبة معرفة من عدّل أو صدّر ماذا
- التباس مسؤولية الدفع والتجديد
- ارتباط أعمال العملاء بحساب واحد

## 5) خدمات وسيطة مبنية على Gamma API

- مفيدة إذا أردت تسعيرًا وحداتيًا أسهل للأعمال
- لكنها تضيف طبقة تكلفة فوق استهلاك اعتمادات Gamma
- راجع مسار البيانات والفواتير وربطها مع `credits.deducted`

وتحقق دائمًا من سياسة تخزين البيانات لدى المزود الوسيط، خصوصًا في المحتوى الحساس.

## 6) إرشاد شراء عملي

- استخدام متقطع: ابدأ Free/Plus وراقب استهلاكك الحقيقي
- إنتاج أسبوعي وتعاون فريق: قارن Pro متعدد الحسابات مع Teams/Business
- أتمتة ثقيلة: استخدم Pro+API وقرّر الترقية حسب بيانات خصم الاعتمادات الفعلية

إذا بدأت بحساب مشترك لتجربة سريعة، ضع من البداية تاريخ انتقال واضح إلى نموذج مقاعد رسمي عند توسع الاستخدام.

### الخلاصة

أفضل خطة ليست الأرخص على الورق، بل التي تجعل تكلفة كل مخرج قابلة للتنبؤ تحت حملك الحقيقي.
