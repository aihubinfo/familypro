买 Gamma 会员时，**每月给多少积分**、**做一次演示大概烧多少分**、**要不要走 API 或第三方按次服务**，会直接决定总成本与协作方式。下面按官方帮助中心与开发者文档整理；**涉及美元标价与加购单价时，会标注数据日期，并强调以结算页为准**。

> 数据日期：2026-04-02  
> 说明：套餐名称、积分数量、扣费规则与美元标价会因地区、税费、促销与产品迭代而变化；下表以 Gamma 公开文档为准，**价格数字若与 [Gamma Pricing](https://gamma.app/pricing) 不一致，一律以官网结算页为准**。

## 1) 各档会员：每月积分与核心差异（官方）

Gamma 在「升级说明」中列出了各档**每月发放的 AI credits**（付费档按月刷新；年付与月付的**每月积分相同**，年付通常为折扣价）。下表为文档中已写明的数量级：

| 档位 | 每月积分（官方） | 与创作相关、且常被拿来对比的要点 |
| --- | --- | --- |
| **Free** | 新用户一次性 **400**（**不按月刷新**） | 单次 AI 生成最多约 **10** 张卡（可再手动加）；基础图片模型等 |
| **Plus** | **1,000** | 单次 AI 生成最多约 **20** 张卡；去导出水印等（见官方功能表） |
| **Pro** | **4,000** | 单次最多约 **50** 张卡；**API 访问**；更全图片模型与站点/分析等（见官方） |
| **Ultra** | **20,000** | 单次最多约 **75** 张卡；**API**；更高级图片模型与更高站点配额等 |
| **Teams** | **6,000 / 席** | 在 Pro 能力之上面向团队：集中计费、共享文件夹、品牌主题等；**至少 2 席**起（定价见官网/团队说明） |
| **Business** | **10,000 / 席** | 在 Teams 之上偏企业能力（如 SSO 等，以官方为准）；**至少 10 席**起 |

团队档标价在官方「Teams 与 Business」说明中给出了**美元参考**（按席、含最低席位数）：**Team 约 $240/席·年（至少 2 席）**、**Business 约 $480/席·年（至少 10 席）**——**是否按月折算、是否含税，以你结账页为准**。

**个人档常见美元月价（公开页面常见标价，仅供参考）**：Plus 约 **$8/月**、Pro 约 **$18/月**、Ultra 约 **$100/月**（同上，**以 [Gamma Pricing](https://gamma.app/pricing) 实时标价为准**）。

### 1.1 积分结转与加购（官方）

- **结转**：未用完的积分可结转，**上限约为「月包大小的 2 倍」**（帮助中心用 Plus **1,000/月**举例：余额最高约 **2,000**；Pro/Ultra/Teams/Business 亦按「2×月包」类推，具体以你账户与文档为准）。
- **加购**：付费用户可购买额外包，常见标价为 **1,500 credits = $6**（**以 [加购说明](https://help.gamma.app/en/articles/12466653-how-do-i-purchase-more-credits) 与结算页为准**）。
- **免费档**：积分用完后不刷新；可通过邀请等获得额外积分（规则见帮助中心）。

## 2) 积分怎么扣：两套口径不要混用

官方存在**两套说明**：一套面向**产品内点按钮**的 AI 行为，一套面向 **API 异步生成**。计费都以 **credits** 为单位，但**不能简单用「网页里 40 分一次」去套 API 的「每张卡 1～3 分」**——因为场景不同。

### 2.1 应用内（帮助中心列出的固定费率）

以下摘自帮助中心「How do credits work in Gamma」，并注明官方提示**费率可能变更**：

| 行为 | 积分（文档表述） |
| --- | --- |
| **用 AI 创建**（Create with AI） | **40 credits / 每个 gamma** |
| **用 AI 新增一张卡片**（Add card with AI） | **5 credits / 张** |
| **与 AI 对话/改写等**（Chat with AI） | **10 credits / 次有效建议**（官方说明**错误澄清不扣**） |
| **AI 配图（按次提示）** | **10 credits / 次图片提示** |
| **Continue with AI**（如 `/continue`、`+++`） | **2 credits / 次** |
| **API** | 见开发者文档与账单 |

**单次「从 0 做一份演示」的粗算思路（界面内）**：

- **只做一轮「AI 创建」、不在此后再 AI 加卡**：下限可记 **≈ 40 credits / 份 gamma**（仍取决于你是否在同一流程里触发其它 AI 步骤）。
- **创建后又用 AI 加了 \(k\) 张卡**：在 40 的基础上大约再加 **\(5k\)**（若还多次对话、配图、Continue，再逐项加）。

官方强调：**实际扣费以账户余额变化为准**；模型与功能迭代会导致规则调整。

### 2.2 API / 异步生成（开发者文档）

开发者文档说明：API 使用 **credit 制**；一次生成完成后，在 **`GET /v1.0/generations/{generationId}`** 的响应里会有 **`credits.deducted` / `credits.remaining`**，应以**该返回值**为最终依据。

文档给出的**量级**（带「内部选模」因素，非固定单价）包括：

- **文本卡片**：约 **每张卡 1～3 credits**（随内部文本生成模型变化）。
- **配图**：随 `imageOptions.model` 等变化；文档示例区间包括**标准约 2～15 / 张**、**高级约 20～33**、**Premium 约 34～75**、**Ultra 档模型约 30～125** 等（均为官方文档中的区间表述）。
- **从模板生成**（`POST /generations/from-template`）可能比标准生成**略贵**。

文档还给了**示意场景**（非承诺）：例如约 **10 张卡 + 5 张基础配图** 的总消耗可能在**约 20～60 credits** 量级；更高阶配图与更多卡会显著上升。

**可靠用法**：若你要写进内部 ROI 表，**请以某次真实调用的 `credits.deducted` 为准**，再取平均，而不是只用网页端的「40/次」去估 API。

## 3) 把「一份 PPT」换算成积分：建议用的估算表

下面给**工程上可用**的估算方式（仍非合同承诺）：

| 场景 | 建议算法 |
| --- | --- |
| **纯网页、从 Create 走一轮 AI 出稿** | 先按 **40 credits** 估底；若你会再 AI 加卡，加上 **5×张数**；对话/配图/Continue 按 §2.1 表累加。 |
| **API 批量生成** | 用文档区间做**区间估算**，上线后用 **`credits.deducted` 校准**；大图、多图、高阶模型取区间上沿。 |
| **Remix / 模板** | Remix 也消耗 credits（见帮助中心）；模板 API 可能略贵，按响应字段为准。 |

**数值 sanity check（界面内粗算）**：若一个月只用「AI 创建」做 **25 份**新 gamma、且每份不再加卡，约 **25×40 = 1,000 credits**，大致贴 **Plus 的月包**——一超出就要加购或升档。（真实世界几乎总会发生加卡、改稿与配图，实际会更高。）

## 4) 「共享账号」vs 正规多席位：产品逻辑与取舍

官方在加购与团队说明中强调：**积分按用户发放，不按团队「共用一池」**；多人工作区里自动加款也可能**按成员分别触发**。

**多人共用同一个付费登录（非按席扩容）时，通常要面对的是**：权限边界不如一人一账号清晰、导出/编辑记录不易按人区分、额度与加购责任需要团队自己约定。是否被服务条款允许，应以 [Gamma Terms](https://gamma.app/terms) 最新版本为准；有合规审计要求时，更稳妥的是走**正式多席位**或企业采购流程。

**更适合考虑「临时共用」的典型情况**（仍建议尽快迁到正式方案）：人数很少、用量不大、预算暂时不足以按席购买，但需要先跑通流程；此时尽量做好**密码与设备管理**、约定**谁负责续费与加购**，避免与客户数据混用同一账号做长期交付。

**正规多席位（Teams/Business）**：有最低席数与更高单价，但对应**分用户积分、管理、品牌主题与协作能力**——适合成员稳定、需要长期协作与统一品牌的工作区。

## 5) 基于 Gamma API 的「按次收费」类工具：能补什么、要核对什么

**常见形态**：在 **Gamma API** 之上做封装或自动化，向业务侧按「每次生成、每张、每个项目」等计费。

**往往更省事的地方**：

- 把异步生成、轮询结果、导出等环节做成**可对接业务系统**的一条链路，减少自己写胶水代码的时间。
- 对非技术同事可以呈现为**固定单价或工单**，便于内部结算（具体取决于供应商产品）。

**仍建议你自己核对的几点**（属于工程常规，不必过度解读成「不可靠」）：

- **底层仍走 Gamma credits**；API 访问通常需要 **Pro/Ultra/Teams/Business** 等具备 API 权限的档位（见开发者文档）。第三方报价 + 你的订阅成本，一起算总账更清楚。
- **数据流**：正文与图片是否经第三方中转、保存多久；若涉及客户资料，按你们公司的数据政策选型即可。
- **扣费对齐**：复杂生成仍以官方返回的 **`credits.deducted`** 与账单为准，用来校准你对「单次成本」的心理预期。

**更适合走这类工具的人群**：已有内部系统/CRM、希望把「做 Deck」嵌进流程；或项目制、需要给业务方**一口价**而不是按 Gamma 月包讲清楚的人。自己能熟练调 API、且用量很低的团队，也可以直接对接官方 API，不必强行上中间层。

## 6) 购买建议（按人群，含共享账号与 API 工具）

**先按用量选档**

- **偶尔做几份对外 Deck**：先用 **Free/Plus** 摸清自己会触发多少「AI 创建 / 加卡 / 配图」；总逼近月包再考虑 **Pro**（积分与单次卡上限更高，且含 **API** 若你要自动化）。
- **每周多份、且要统一品牌/协作**：评估 **Teams/Business** 是否比「多个个人 Pro」更省管理成本；注意**最低席数**与**每席积分**。
- **重度自动化（批量从系统生成）**：优先 **Pro 及以上 + API**，用 **`credits.deducted` 做预算**；**Ultra** 更适合**高卡数 + 高阶配图**或更大月包。

**共享账号更适合谁、不更适合谁**

- **更适合**：人数极少、短期试水、预算暂时按不了席，但需要先统一用一个账号把模板与风格跑通的小团队——同时最好规划**何时升级到按人账号**，避免与客户项目长期绑在同一个登录上。
- **不更适合**：成员多、客户多、需要按人追责或审计的组织——应直接看 **Teams/Business** 或明确的一人一账号采购。

**Gamma API / 按次计费类工具更适合谁**

- **更适合**：要把生成嵌进现有系统、希望业务侧按次付费或按工单结算、或不想从零维护轮询与导出链路的团队。
- **更适合自己直连官方 API**：用量不大、有开发资源、希望链路最短、对数据路径完全自控的团队。
